# systemPatterns — Архитектура и ключевые решения

## Главный шов: Command → Simulation → Events

Единственное правило, которое делает возможным будущий онлайн:

```
Указатель/клик ──► Command ──► SimHost ──► Simulation (ECS) ──► Events ──► Renderer + HUD
                                              │                            (только чтение)
                                       авторитетное состояние
```

- **Ввод никогда не меняет состояние игры.** Клик порождает `Command` (намерение), например
  `DetonateCommand{dir, yield}` — не мутацию.
- **Симуляция — единственное, что мутирует состояние.** Она потребляет команды на каждом тике
  и выпускает `Events` (факты): `ExplosionStarted`, `CityHit{deaths}` и т.п.
- **Рендер и UI — только читатели** состояния и событий. Они не трогают логику.

### Почему это про сеть

`SimHost` — интерфейс с формой `postCommand()` / `onEvent()`. Границы Web Worker и сети имеют
**ту же форму**, что и этот шов, поэтому реализацию можно подменить, не трогая вызывающий код:

- `LocalSimHost` — симуляция в основном потоке (этап 1, сейчас).
- `WorkerSimHost` — симуляция в отдельном потоке (Web Worker) — позже, чтобы логика не роняла FPS.
- `NetworkSimHost` — симуляция на сервере — позже, для онлайна.

## Слоистая структура и направление зависимостей

```
src/
  main.ts               bootstrap + игровой цикл
  core/    GameLoop, EventBus (типизированный), time (seeded RNG, часы)
  sim/     Simulation, SimHost (интерфейс + LocalSimHost),
           commands.ts, events.ts, geo.ts (lat/lon ↔ вектор, тайл-математика), cities.ts
  ecs/     world (miniplex), components.ts, systems/ (Missile, Explosion, Casualty)
  render/  Renderer (интерфейс), backend/createRenderer (WebGPU + откат),
           Scene, GlobeView, TileLayers, ExplosionView, DecalView, effects/particles
  input/   PointerController, CameraRig
  ui/      Hud, styles.css
  assets/  config.ts (мощности зарядов, URL тайлов, константы тюнинга)
test/      sim/ (Vitest, чистая логика)
```

**Жёсткое правило:** `sim` и `ecs` НЕ импортируют ничего из `render`/`ui`/`input`.
Они чистые и тестируются headless (без GPU).

## Симуляция

- **Фиксированный таймстеп** (30 Гц) с аккумулятором; рендер интерполирует между состояниями
  симуляции — плавность не зависит от FPS.
- **Seeded RNG** в `core/time` — никакого `Math.random()` в симуляции — прогоны воспроизводимы
  (нужно для реплеев и netcode).
- **ECS: miniplex** — эргономичный TS, компоненты-объекты, тривиальная сериализация в JSON
  (пригодится для снапшотов состояния по сети). Сотни сущностей — с запасом в его рамках.
  Компоненты вроде `OnSphere{dir}`, `Warhead{yield,seed}`, `Blast{age}`, `City{name,pop,alive}`.
  Системы — чистые функции `(world, dt) → events`.

## Рендеринг

### Абстракция рендерера
- `three/webgpu` `WebGPURenderer` с **автоматическим откатом на WebGL2**, спрятан за интерфейсом
  `Renderer` (`init/resize/render/dispose/capabilities`). Игровая логика зависит от интерфейса,
  не от конкретного класса.
- Материалы и эффекты пишем на **TSL** (Three Shading Language, узловые материалы) — они
  компилируются и в WGSL (WebGPU), и в GLSL (WebGL2). Один код на оба бэкенда.
- **Частицы взрыва — инстансированные меши, анимированные временем в вершинном шейдере**
  (работают одинаково на обоих бэкендах). Настоящие compute-шейдеры бережём на будущее для
  тяжёлых симуляций, где WebGPU реально выигрывает — чтобы не иметь хрупкого compute-only пути,
  ломающегося на откате WebGL2.

### Оптимизация (устранение лагов демо)
Причины тормозов в демо и их лечение:
- **~150 отдельных `Sprite` на взрыв** (150 draw-call) → **GPU-инстансинг**: один инстанс-меш на
  тип эффекта, тысячи частиц за **один draw-call**; движение считается в шейдере из `(spawnTime,
  seed)` — ноль работы CPU на кадр, ноль JS-объектов на частицу.
- **`PointLight` на каждый взрыв** — добавление света в three.js **заставляет все освещённые
  материалы перекомпилировать шейдер** (главный рывок) → **убираем динамический свет**: вспышка
  становится additive/bloom-эффектом, остаётся один статический «солнечный» свет.
- **Декали-кратеры копятся без предела** → **пулинг + лимит** (~512 кратеров, старые
  переиспользуются).
- **Всё эфемерное — из пулов**: взрывы, ракеты, декали заранее аллоцированы, ничего не
  создаётся/собирается GC во время игры.

## Материал поверхности и разрушаемость

- **Материал — авторитетный факт sim.** `sim/material.ts::materialAt(lon,lat)` (чистая, headless)
  классифицирует точку в `Surface` (land/water/ice) + `Biome` из грубой запечённой маски реальных
  материков (`sim/landmask.ts`, 512×256, генератор `scripts/gen-landmask.mjs`) и правил широтных
  поясов+шум. Событие `explosionStarted` несёт `surface`/`biome` — рендер читает, не вычисляет.
  Готово к онлайну/реплеям; геймплей (жертвы) материал пока не меняет.
- **Поле урона = GPU displacement.** `render/DamageField.ts` — накопительное equirect render-target
  (R=глубина воронки, G=гарь, B=оплавление). Детонация делает разовый **splat** воронки (не на кадр);
  накопление — **ping-pong max** (два RT, `max(prev, stamp)` в шейдере) вместо blend-equation MAX
  (тот НЕ поддерживается WebGL2-бэкендом three — важный урок). У полюсов вес по долготе
  масштабируется `cos(lat)=sin(v·π)`, иначе воронка размазывается в «звезду». Глобус — плотная сетка
  384×192; вершинный TSL-шейдер вдавливает поверхность по полю, фрагментный чернит/оплавляет
  (полынья на льду). Сброс = очистка обоих RT.
- **Взрыв маршрутизируется по `surface`** в `render/Scene.ts`: суша/лёд → гриб+splat в поле; вода →
  `render/WaterBurstView.ts` (подводный: столб/купол/кольцо, в поле НЕ пишет — следа нет). Общая
  геометрия купола — `render/effects/geometryUtils.ts`.
- **Dev-инструменты — за флагом, не удаляются.** Debug-хуки `__strike/__reset/__lookAt`
  (`src/debug/devHooks.ts`) под `import.meta.env.DEV` (вырезаются из прод-бандла); headless-харнесс
  приёмки — `scripts/accept/shots.mjs` (`npm run accept`).

## Тестирование
- **Vitest** для логики симуляции (гео-математика, расчёт жертв, ECS-системы, `materialAt`/landmask/
  `dirToLonLat`) — детерминировано, без GPU.
- Рендер проверяем методом **headless Chrome + скриншот** (`scripts/accept/shots.mjs`): WebGL2
  swiftshader, сбор консоли (ошибки компиляции шейдеров — фейл), скриншоты ключевых состояний.
  TSL-шейдеры компилируются в браузере в рантайме — сборка/тесты их не ловят, только харнесс.
