# Вода-жидкость: анимированный океан + интерактивное поле — дизайн

_Дата: 2026-07-05 · Ветка: `feat/liquid-ocean-water` · Статус: согласован (мокап утверждён), готов к плану_

## 1. Проблема и цель

Сейчас океан — **статичная синяя заливка** на биом-текстуре глобуса. Вода не двигается, не
отражает, никак не реагирует на удар — «плоская картинка», а не жидкость. Это первый из трёх
подпроектов большой задачи «реализм воды и суши» (см. §9 «Дорожная карта»).

**Цель подпроекта №1:** сделать океан **жидкостью** — всегда анимированная волновая поверхность
+ **интерактивный отклик** на возмущения (клик/удар): рябь расходится, каверна от удара
раскрывается и смыкается в кольцо. Это же интерактивное поле станет фундаментом для реалистичного
водного взрыва (подпроект №3).

Утверждённый визуальный эталон — интерактивный мокап (подход **A**), одобрен пользователем:
анимированные волны, Френель-отражение неба, цвет по глубине, пена на гребнях/у берега,
клик-рябь, каверна от удара.

### Граница скоупа (YAGNI)

- **Только океан-визуал.** Геймплей (жертвы/урон) **не меняется** — как и в фиче материала, вода
  влияет лишь на картинку и события.
- **Не делаем** в №1: буйантность/плавающие объекты/корабли; FFT-океан (Tessendorf); настоящие
  screen-space отражения (SSR) — только аналитическое небо; реалистичный водяной столб взрыва
  (это №3 — здесь вода лишь **реагирует** полем, столб пока остаётся текущий `WaterBurstView`);
  LOD/детализация суши (это №2); глубина воды из данных батиметрии (мелко/глубоко берём из
  расстояния до берега, не из реальных глубин).

## 2. Согласованные решения (из брейншторма)

| Вопрос | Решение |
|--------|---------|
| Подход | **A** — анимированная водная оболочка + интерактивное height-field, отклонены B (полный FFT — тяжело/плитка на сфере) и C (без интерактива — не закрывает удар) |
| Волны | **Gerstner-свеллы (макро, вершинный сдвиг) + 3D-шум (микро, возмущение нормали)**. 3D-шум по позиции — нет растяжения у полюсов |
| Шейдинг | **Ручной TSL на `MeshBasicNodeMaterial`** (как атмосфера в `GlobeView`), **без динамического света** — константный «солнце»-юниформ |
| Отражение | **Аналитическое небо** (Френель → градиент неба + блик солнца). SSR/env-cubemap — бэклог |
| Интерактив | **GPU ping-pong height-field** (волновое уравнение), структура — порт паттерна `DamageField` |
| Формат поля | `RGBA16F` (знаковые высота/скорость). Фолбэк — упаковка в `RGBA8`, если нет `EXT_color_buffer_float` |
| Маска океана | **Equirect coast-field текстура** (океан/суша + расстояние до берега), запекается из существующего `landmask` при старте |

## 3. Архитектура

Шов `Command → Simulation → Events → Renderer` сохраняется. Новое — только на стороне рендера
плюс подписка на существующее событие `explosionStarted` (несёт `surface`).

```
клик по воде (input) ─┐
                      ├─→ WaterField.splat(dir, strength, radius)   (рябь/удар вписывают импульс)
explosionStarted{     │
  surface:'water'} ───┘
                                   каждый кадр
WaterField.step(dt) ──ping-pong──→ поле высот (RGBA16F equirect)
                                          │ .texture
CoastField (запечён при старте) ──────────┤
константный «солнце»-юниформ ─────────────┤
                                          ▼
                                     OceanShell  (сфера r≈1.0008, discard на суше)
                                        макро-волны: вершинный Gerstner + сэмпл поля
                                        микро-детали: возмущение нормали 3D-шумом + градиент поля
                                        цвет: глубина + Френель-небо + блик + пена
                                          ▼
                                       рендер поверх глобуса
```

### Компоненты (изолированные, у каждого одна задача)

**`render/WaterField.ts` — интерактивное волновое поле.**
Порт структуры `DamageField`: две `RenderTarget` (ping-pong), фиксированный шаг симуляции.
- Хранит на пиксель: высоту и скорость (каналы R/G у `RGBA16F`).
- `step(dt)`: один-два подшага волнового уравнения `a = c²·∇²h − damping·v`, интегрирование
  Верле/полу-неявное; читает предыдущий RT, пишет в текущий (как `prevRt`/`rt` в `DamageField`).
  Лапласиан — 4-соседний сэмпл с `cos(lat)`-поправкой шага по долготе (та же поправка, что в
  `DamageField`, комментарий про полюса). Шов долготы — `RepeatWrapping`.
- `splat(dir, strength, radius)`: впечатывает гауссов импульс в скорость/высоту в точке `dir`
  (маппинг `dirToLonLat` → uv, как `DamageField.splat`). Рябь — малый `strength`; удар —
  большой отрицательный (каверна) с последующим упругим кольцом от волнового уравнения.
- `clear()` на `planetReset`.
- `texture`: стабильная идентичность `.texture` (захватывается `OceanShell` один раз).
- **Затухание** гарантирует, что поле само возвращается к штилю — постоянного следа нет
  (согласуется с «вода смыкается»).

**`render/CoastField.ts` (или функция-билдер) — маска океана и расстояние до берега.**
Запекает при старте equirect-текстуру из `sim/landmask`:
- канал: суша/океан (0/1) + мягкое **расстояние до берега** (для мелководного цвета и береговой
  пены). Расстояние — дешёвый разлив (несколько итераций max-фильтра) по маске, один раз на CPU.
- Разрешение ~1024×512 (как биом). Возвращает `THREE.DataTexture`/`CanvasTexture`.

**`render/OceanShell.ts` — анимированная водная поверхность.**
- Меш: `SphereGeometry(R_OCEAN, …)`, `R_OCEAN≈1.0008` (чуть выше глобуса — избегаем z-fighting с
  ocean-цветом глобуса; на стратегическом зуме разница невидима). Тесселяция достаточна для
  макро-волн (переиспользуем масштаб `GLOBE_LON/LAT_SEG` или чуть ниже).
- Материал `MeshBasicNodeMaterial`, всё освещение — вручную на TSL:
  - **Вершина:** сдвиг вдоль нормали = сумма 2–4 Gerstner-свеллов (анимация по `uTime`) +
    выборка `WaterField` (макро-отклик удара/каверны).
  - **Discard на суше:** по `CoastField` (`opacityNode`/alpha-clip) — оболочка рисуется только над
    океаном; на суше виден глобус.
  - **Фрагмент:** нормаль воды = возмущение геонормали 3D-шумом (fbm 3 октавы, анимация) +
    градиент `WaterField`; цвет = смесь глубокий/мелкий по расстоянию до берега; Френель →
    небо-градиент; блик по константному солнцу (Blinn-Phong, высокий показатель); пена = гребни
    (порог высоты/крутизны) + береговая полоса.
  - `transparent`/`depthWrite` — подобрать так, чтобы корректно ложиться на глобус и атмосферу.

**Wiring — `render/Scene.ts`.**
- Конструирует `WaterField`, `CoastField`, `OceanShell` (в `spinGroup`, чтобы вращаться с глобусом).
- Каждый кадр: `waterField.step(dt)`; продвигает `uTime` океана.
- На `explosionStarted` c `surface==='water'`: `waterField.splat(dir, strengthByYield, radiusByYield)`
  (в дополнение к текущему `waterBurstView.spawn` — тот пока не трогаем).
- Клик по океану (input): `waterField.splat(dir, малый, малый)` — «живость» от взаимодействия
  (опционально; основная цель — отклик на удар).
- `planetReset`: `waterField.clear()`.

## 4. Поток данных

1. Событие удара/клик → `WaterField.splat` пишет импульс в поле (equirect).
2. `WaterField.step` каждый кадр эволюционирует поле (ping-pong), затухание возвращает к штилю.
3. `OceanShell` сэмплирует поле (вершина: высота для макро-сдвига; фрагмент: градиент для нормали)
   + собственные анимированные Gerstner/шум-волны → рисует жидкую поверхность.
4. `CoastField` статически указывает шейдеру, где океан (discard суши) и как близко берег
   (мелководный цвет + пена).

## 5. Обработка ошибок / краевые случаи

- **Нет `EXT_color_buffer_float` (WebGL2):** фолбэк — упаковка высоты/скорости в `RGBA8`
  (кодирование фикс. точкой) или деградация до **без-интерактивного** режима (только Gerstner/шум),
  чтобы приложение не падало. Определяем поддержку при старте, ветвимся один раз.
- **Полюса:** `cos(lat)`-поправка шага (как `DamageField`); импульсы у самых полюсов редки —
  допускаем лёгкий артефакт растяжения ряби.
- **Шов долготы:** `RepeatWrapping` на RT и текстурах — рябь бесшовно переходит через ±180°.
- **Пул исчерпан / поле насыщено:** волновое уравнение с затуханием естественно ограничивает
  амплитуду; отдельного пула нет.
- **Совместимость бэкендов:** избегаем `CustomBlending+MaxEquation` (падает на WebGL2 —
  задокументировано в `DamageField`); все проходы — обычный опаковый вывод с чтением prev-RT.

## 6. Производительность

- **Симуляция поля:** 1–2 маленьких draw/кадр на `RGBA16F` ~1024×512 — дёшево (как splat
  `DamageField`, только каждый кадр вместо разового).
- **`OceanShell` фрагмент** — главная статья: fbm по всем пикселям океана каждый кадр. Держим
  **3 октавы**, при удалении камеры — меньше (LOD по дистанции). Макро-волны — вершинные (дёшевы).
- **Никакого динамического света** — соблюдаем ключевое ограничение движка (устранённый источник
  лагов эталона).
- Цель — **60fps** на среднем железе при обычном зуме; замер в приёмке.

## 7. Тестирование

**Визуальную приёмку не автоматизируем** (headless screenshot-harness слишком медленный) —
визуальную часть проверяет пользователь вручную в браузере и сообщает правки. Автоматом гоняем
только логику/типы/сборку:

- **Unit (headless, чистые функции):** маппинг `dir→uv` поля; детерминизм `CoastField` (одинаковый
  вход → одинаковая маска); затухание поля к нулю за конечное время (симуляция шага на CPU-эталоне
  волнового уравнения); корректность `cos(lat)`-поправки шага.
- **Сборка/типы:** `tsc` без ошибок; сборка проходит.
- **Smoke без визуала:** приложение стартует, консоль **без ошибок шейдеров на WebGL2** (быстрый
  headless-запуск без скриншот-диффов).
- **Визуально (вручную, пользователь):** анимация океана, Френель/цвет/пена, клик-рябь, каверна от
  удара смыкается, чёткость берега, перф/FPS на целевом зуме.

## 8. Влияние на существующий код

- **`GlobeView`:** без изменений формы; ocean-цвет глобуса остаётся как подложка (скрыт оболочкой,
  виден как fallback, если интерактив отключён). Возможна мелкая правка порядка рендера.
- **`WaterBurstView`:** **не трогаем** в №1 (переработка столба — подпроект №3). Здесь он лишь
  сосуществует с новым полем.
- **`Scene.ts`:** добавляет конструирование и wiring трёх новых компонентов.
- **`config.ts`:** новые константы (разрешение поля, `R_OCEAN`, коэффициенты волн/затухания,
  strength/radius по мощности).

## 9. Дорожная карта (контекст, не в этом скоупе)

1. **№1 — вода-жидкость** ← этот документ.
2. **№2 — LOD-детализация суши** (устранить «мыло»/шакальность при зуме: chunked-LOD/clipmaps,
   стриминг тайлов, detail-карты). Отдельная спека.
3. **№3 — реалистичный водный взрыв** (spray dome / полый столб / обрушение / base surge; капли —
   баллистические GPU-частицы; **втыкается в `WaterField` из №1**). Отдельная спека.

## 10. Открытые вопросы (уточнить при плане)

- Точное `R_OCEAN` и стратегия против z-fighting (радиус-офсет vs `polygonOffset`).
- Разрешение поля (512×256 достаточно? 1024×512?) — баланс качество ряби / перф.
- Клик-рябь по океану включаем в №1 или только отклик на удар (минимизировать скоуп).
- Источник «солнце»-направления: переиспользовать существующий у сцены или новый константный
  юниформ.
