# Детализация поверхности на зуме (2B) — дизайн

_Дата: 2026-07-05 · Ветка: `feat/terrain-detail-lod` · Статус: автономно утверждён (пользователь делегировал решения)_

## 1. Проблема и цель

При приближении камеры суша превращается в **крупные блоки/«мыло»**: биом-текстура — equirect
1024×512 с поблочной классификацией биомов (`MaterialGlobe.buildBiomeCanvas` → `materialAt` на
тексель), поэтому вблизи каждый тексель (~0.35° ≈ десятки км) читается как большой цветной блок,
а магнификация даёт замыленность/пикселизацию. Детали не появляются.

**Цель:** при приближении поверхность обретает **процедурную детализацию** — мелкая, чёткая
вариация цвета и рельефа (нормали), нарастающая по мере приближения камеры. Далёкий вид
(стратегический зум) остаётся как сейчас; вблизи — «мыло»/блоки заменяются хрустящей деталью.

### Подход (и почему не CDLOD/тайл-стриминг)

Полноценный terrain-LOD (chunked quadtree / CDLOD / стриминг тайлов высот и снимков, как CesiumJS)
— **отвергнут для этого этапа**: глобус стилизованный (нет спутниковых снимков для стриминга),
это движковый мега-рефактор (гигабайты данных, воркеры, кэш тайлов) ради того, чего у нашей
стилизованной планеты нет. Берём **процедурную детализацию** (detail albedo + detail normal из
многооктавного шума, biome-aware, сила по дистанции камеры) — бесконечная детализация без
дополнительных данных/памяти. Это ровно приём «detail-текстуры/процедурная детализация без
гигабайт» из исследования. CDLOD/реальная гео-детализация — в бэклог (если появится реальный
геослой).

### Граница скоупа (YAGNI)

- Только визуал суши на зуме. Геймплей/sim не трогаем.
- **Не делаем:** chunked-LOD/квадродерево/стриминг тайлов; реальные снимки/высоты; геометрический
  LOD (сетка глобуса остаётся 384×192); детализацию воды (у неё свой шум, подпроект №1) и кратеров
  (2A, свой микрорельеф — но общий шум переиспользуем). Батиметрию/облака — нет.

## 2. Опорная архитектура

- `render/GlobeView.ts` — `MeshPhongNodeMaterial`, `colorNode` (биом → гарь/зоны кратера → полынья),
  `normalNode` (микрорельеф кратера через `mix(materialNormal, detail, mask)`), `bumpMap`=топо.
  Сюда добавляется detail-слой суши.
- `render/OceanShell.ts` и `render/GlobeView.ts` содержат **дублирующиеся** `hash/noise/fbm` (TSL
  `Fn`) — выносим в общий модуль (закрывает DRY-бэклог 2A).
- Дистанция камеры в шейдере: `length(cameraPosition − positionWorld)` (оба узла уже используются в
  `GlobeView.buildAtmosphere`). Камера-дистанция из `CameraRig` ~1.6..6 (радиус планеты 1).

## 3. Согласованные решения (автономно)

| Вопрос | Решение |
|--------|---------|
| Метод | **Процедурная детализация** (шум), НЕ тайл-стриминг |
| Что детализируем | **Albedo** (мелкая вариация цвета вокруг биом-цвета) + **normal** (микрорельеф суши) |
| Управление силой | **По дистанции камеры**: далеко → 0 (как сейчас), близко → полная деталь. `smoothstep(DETAIL_FAR, DETAIL_NEAR, dist)` |
| Biome-aware | Характер детали зависит от биом-цвета/типа (амплитуда/оттенок вариации от base) |
| Общий шум | Вынести `hash/noise/fbm` в `render/noise.ts`, переиспользовать в OceanShell/GlobeView |
| Не на воде | Detail суши применять только к суше (по landmask/биому — не поверх океан-цвета, который скрыт водной оболочкой) |
| Композиция с кратером | Detail-нормаль и кратер-микрорельеф складываются согласованно (обе через `normalNode`, суммируются до `mix` с `materialNormal`) |

## 4. Архитектура и компоненты

**`render/noise.ts` (новый).**
- Экспорт TSL-`Fn`: `hash(p)`, `noise3(p)`, `fbm(p, octaves)` — идентичны текущим в OceanShell/GlobeView
  (перенос без изменения поведения). Параметр октав. Один источник истины шума на рендер.

**`render/GlobeView.ts` (модификация).**
- Импортировать шум из `noise.ts`; удалить локальные дубли (и в OceanShell — тоже перевести на общий).
- **Detail-strength по дистанции:** `detailK = smoothstep(DETAIL_FAR, DETAIL_NEAR, dist) · onLand`,
  где `dist = length(cameraPosition − positionWorld)`, `onLand` — маска суши (по биому/landmask,
  чтобы не детализировать под-водную часть глобуса).
- **Detail albedo:** к биом-цвету добавить мелкую многооктавную вариацию (высокая частота по
  `positionLocal`), масштаб/оттенок от base (biome-aware): `color = base · (1 + (fbm−0.5)·amp·detailK)`
  + лёгкий сдвиг к светлым/тёмным пятнам. Не менять цвет вдали (detailK→0).
- **Detail normal:** микрорельеф суши по высокочастотному шуму, сила ∝ `detailK·DETAIL_NORMAL_STR`.
  Сложить с кратер-микрорельефом (2A) в едином возмущении нормали (оба до `mix(materialNormal,…)`).
- Порядок: detail НЕ перекрывает зоны кратера/полынью (кратер — поверх/сильнее по своей маске).

**`render/OceanShell.ts` (модификация, минимальная).**
- Перевести на `noise.ts` (убрать дубль). Поведение воды не меняется (те же формулы).

**`config.ts` — константы:** `DETAIL_NEAR`, `DETAIL_FAR` (дистанции рампы), `DETAIL_ALBEDO_AMP`,
`DETAIL_NORMAL_STR`, `DETAIL_FREQ`, `DETAIL_OCTAVES`.

## 5. Поток данных

1. Фрагмент суши глобуса → считает дистанцию до камеры → `detailK` (0 далеко, 1 близко).
2. Albedo = биом-цвет + процедурная вариация · detailK; Normal = геонормаль + микрорельеф · detailK
   (сложенный с кратер-микрорельефом 2A).
3. Вдали `detailK≈0` → вид как сейчас; вблизи блоки/мыло заменяются деталью.

## 6. Обработка ошибок / краевые случаи

- **Дистанция/зум:** `DETAIL_NEAR/FAR` подобрать по диапазону `CameraRig` (зум 1.6..6). Клампить
  `detailK` в [0,1].
- **Швы/полюса:** шум по `positionLocal` (3D) — нет seam/полюсных артефактов (как в OceanShell/2A).
- **Не поверх воды:** маска суши гарантирует, что detail не лезет на океан-цвет глобуса (скрыт
  оболочкой, но корректность).
- **Композиция с кратером:** detail-нормаль складывается, но кратер-зоны цвета (стекло/порода)
  доминируют по своей маске — detail их не «зашумляет» в ноль.
- **Рефактор шума:** перенос в `noise.ts` — поведение байт-в-байт; существующие тесты воды/кратера
  зелёные, вид воды не меняется.

## 7. Производительность

- **Главный риск:** глобус-фрагмент уже считает fbm кратер-микрорельефа (2A). Detail добавляет ещё
  fbm. Меры: (а) **переиспользовать один вызов шума** для albedo и normal, где возможно; (б) держать
  `DETAIL_OCTAVES` малым (2–3); (в) `detailK` гейтит только результат, но при желании — общий гейт
  по (кратер+detail) активности. Перф-приёмку/финальную настройку октав делает пользователь.
- Никакого динамического света; сетка не меняется (нет доп. геометрии/аллокаций).
- Цель 60fps на обычном зуме; вблизи допускается более тяжёлый фрагмент (меньше площадь экрана —
  меньше фрагментов планеты далёкого фона).

## 8. Тестирование

Визуальную приёмку **не автоматизируем** (правило проекта) — вид/перф проверит пользователь. Автоматом:
- **Unit (vitest):** `detailStrength(dist)`-подобный чистый хелпер (если вынесем рампу дистанции в
  чистую функцию) — 0 далеко, 1 близко, монотонность, кламп. Рефактор шума: существующие тесты
  (`craterProfile`, `ejecta`, вода) остаются зелёными.
- **Сборка/типы:** `tsc` чисто; `lint` чисто.
- **Smoke:** старт без ошибок шейдеров.

## 11. Открытые вопросы (решить в плане, автономно)

- Маска суши в шейдере: по биом-цвету (океан-цвет из config) или по отдельной landmask-текстуре
  (как CoastField в №1). Предпочесть: переиспользовать **CoastField**-подобную маску если дёшево,
  иначе — по биом-принадлежности. Решить в плане (по умолчанию: маска по `materialGlobe`/биому,
  без новой текстуры, если возможно; иначе CoastField).
- Рампа дистанции — чистый хелпер (тестируемый) vs инлайн в шейдере: вынести хелпер-константы, но
  сам `smoothstep` в TSL; тестировать числовой аналог рампы.
- Biome-aware характер: минимально — единый шум с амплитудой от яркости base; расширение по типам
  биомов — бэклог.
